## Why

- **Style learning needs examples + a format signal.** The mature-repo system prompt (§17.1) shows the model up-to-20 recent commit messages and tells it to match their STYLE but never reuse phrasing. `FetchExamples` produces exactly that example list (trimmed, capped) AND the `hasMultiline` flag that picks the conditional multi-line rule in §17.1. This is the single input the prompt builder (S2) consumes.
- **Multi-line detection is the proven awk heuristic, ported faithfully but bug-fixed.** reference_impl.md §6 splits the raw `---%n%B` stream on `---` separators and flags any group with >1 non-empty line as multi-line. The reference awk has a latent bug: its `END` block never re-checks the final accumulated group, so a multi-line OLDEST commit (last group, no trailing `---`) is MISSED (verified empirically — prints 0). The task contract says "if ANY group has >1 non-empty line", so the Go port checks ALL groups incl. the last. This must be an explicit, documented decision so a reviewer doesn't "fix" it back toward the buggy script.
- **Decoupled from git by an interface (plan_overview: M4 is decoupled; builders take scalar settings).** The prompt package must not import `internal/git` in production — that would weld the prompt layer to the git wrapper and block parallel/stub development. A consumer-defined `HistoryReader` interface (`*git.Git` satisfies it structurally) is the seam: M4 depends on git history only as a typed contract, and M6 (generate) injects the real `*git.Git`.
- **The RAW stream is trimmed HERE (not in git).** The dependency `git.RecentMessages` returns the raw `---%n%B` output VERBATIM (blank lines and `---` separators intact) precisely so this layer can run `sed '/^$/d' | head -100` and the awk scan. FetchExamples owns the trim + cap + group-split; do NOT push trimming back into git.

3. **Constants** — `const DefaultExampleCount = 20` (PRD FR11 / reference `-20`; the canonical `n` callers pass) and `const exampleLineCap = 100` (PRD FR11 / reference `head -100`; unexported, internal).
4. **FetchExamples** — `func FetchExamples(g HistoryReader, n int) (examples []string, hasMultiline bool, err error)`:
   - `count, err := g.CommitCount()`; if `err != nil` → `return nil, false, err` (propagate; do NOT swallow — a real git failure is a genuine error, unlike the unborn case which CommitCount already turns into `(0,nil)`).
   - if `count <= 1` → `return nil, false, nil` (new-repo / root-commit path, FR39; NOT an error). This is the gate that §17.2's conventional-commit fallback hinges on.
   - `raw, err := g.RecentMessages(n)`; if `err != nil` → `return nil, false, err`.
   - `groups := splitExampleGroups(raw)` — split into per-commit groups (see #5).
   - Compute `hasMultiline`: `for _, grp := range groups { if len(grp) > 1 { hasMultiline = true; break } }` — over ALL groups, BEFORE the cap (contract: "split the raw log").
   - Build `examples` newest-first with the 100-line whole-message cap: `var total int; for _, grp := range groups { if total+len(grp) > exampleLineCap { break }; examples = append(examples, strings.Join(grp, "\n")); total += len(grp) }`. Groups appear newest-first in `raw` (git log emits newest-first), so iterating in order yields newest-first examples.
   - `return examples, hasMultiline, nil`.
   - godoc cites PRD FR10–FR12, §17.1, reference_impl.md §6, and the (count<=1 ⇒ [],false,nil) new-repo contract.
5. **splitExampleGroups** — unexported helper `func splitExampleGroups(raw string) [][]string` porting the awk/sed pipeline. For each `line` in `strings.Split(raw, "\n")`: if `strings.TrimSpace(line) == "---"` → flush the current group (append to groups if non-empty, reset) and continue (the `---` separator from `--format=---%n%B`); else if `strings.TrimSpace(line) == ""` → skip (the `sed '/^$/d'` blank-line trim); else append the line VERBATIM (NOT trimmed — faithful to sed which only strips empty lines, not interior whitespace) to the current group. AFTER the loop, flush the final group (the OLDEST commit has no trailing `---` — this flush is the deliberate fix for the reference awk's END-block omission; do NOT skip it). Returns groups newest-first. A commit body line that happens to be exactly `---` is treated as a separator (same as the awk `/^---$/`) — note this known, faithful quirk.
6. Edge cases handled naturally: empty `raw` (e.g. n<=0, though callers pass 20) → no groups → `([], false, nil)`; an all-blank group → dropped; a single non-blank line group → counted, not multi-line. No special-case code needed beyond the algorithm above.

### Implementation Patterns & Key Details
```go
func FetchExamples(g HistoryReader, n int) (examples []string, hasMultiline bool, err error) {
    count, err := g.CommitCount()
    if err != nil {
        return nil, false, err
    }
    if count <= 1 {
        return nil, false, nil 
    }
    raw, err := g.RecentMessages(n)
    if err != nil {
        return nil, false, err
    }
    groups := splitExampleGroups(raw)
    for _, grp := range groups {
        if len(grp) > 1 {
            hasMultiline = true
            break
        }
    }
    var total int
    for _, grp := range groups {
        if total+len(grp) > exampleLineCap {
            break
        }
        examples = append(examples, strings.Join(grp, "\n"))
        total += len(grp)
    }
    return examples, hasMultiline, nil
}
func splitExampleGroups(raw string) [][]string {
    var groups [][]string
    var cur []string
    flush := func() {
        if len(cur) > 0 {
            groups = append(groups, cur)
            cur = nil
        }
    }
    for _, line := range strings.Split(raw, "\n") {
        switch {
        case strings.TrimSpace(line) == "---": 
            flush()
        case strings.TrimSpace(line) == "": 
        default:
            cur = append(cur, line) 
        }
    }
    flush() 
    return groups
}

```
