# PRP — P1.M3.T4.S2: internal/git/stage.go — HasStagedChanges / AddAll

## ⚠️ CONTRACT CORRECTION — READ FIRST (the work-item contract inverts the git exit codes)

The work-item contract states: *"HasStagedChanges() (bool, error) — `git diff --cached --quiet`; exit 0 → true (staged), exit 1 → false (clean), other → error."*

**That exit-code mapping is empirically FALSE and must be INVERTED.** `git diff --cached --quiet` follows git's standard `--exit-code` convention — verified four ways on git 2.54.0:

| repo state | command result |
|---|---|
| unborn repo, nothing staged | **exit 0** (clean) |
| unborn repo, file staged (`git add`) | **exit 1** (staged) |
| committed repo, clean index | **exit 0** (clean) |
| committed repo, modified+staged tracked file | **exit 1** (staged) |

git's own `--exit-code` help: *"exit with 1 if there were differences, 0 otherwise"* (`--quiet` implies `--exit-code`). PRD **FR16** corroborates: *"If `git diff --cached --quiet` reports no staged changes ..."* ⇒ exit 0 = no staged. The method NAME `HasStagedChanges` must return `true` when staged changes exist (= exit 1). Implementing the literal (inverted) contract would make `HasStagedChanges` return `true` on a CLEAN index — the opposite of its name — which would break the CLI empty-check (P1.M7.T2.S1) and the auto-stage path (P1.M7.T2.S2).

**This PRP implements the CORRECTED, empirically-verified semantics:**
- exit 0 → `return (false, nil)` (clean index, NO staged changes)
- exit 1 → `return (true, nil)` (staged changes present)
- any OTHER non-zero exit → `return (false, err)` (the typed `*ExitError`; genuine failure)

Full evidence + resolution in `plan/001_f1f80943ac34/prps/research/P1_M3_T4_S2_research.md`.

## Why

- **These are the staging seam the CLI layer needs.** The v2-readiness design (decisions.md §1) pins the core as `CommitStaged(ctx, cfg)` that ASSUMES the index is already staged, with staging policy in the CLI layer (`main` = `maybeAutoStage(); CommitStaged()`). `HasStagedChanges` is the empty-check (FR5/FR16: "if `git diff --cached --quiet` reports no staged changes"); `AddAll` is the `git add -A` step of the auto-stage path (FR16) and the `--all`/`-a` force-stage (FR20). Keeping them as bare primitives (no policy) here is what lets the CLI layer decide auto-stage vs exit-2 cleanly and keeps the generate layer staging-agnostic.
- **The exit code IS the signal — do NOT treat exit 1 as an error.** `git diff --cached --quiet` deliberately uses exit status as a boolean (exit 1 = differences). The shipped `run` seam wraps a non-zero exit in a typed `*ExitError{Code}`; HasStagedChanges must `errors.As` into it and read `Code == 1` as the normal "staged" signal (return `(true, nil)`), NOT surface it as a failure. This mirrors how plumbing.go's `RevParseHEAD`/`WriteTree` and log.go's methods each `errors.As` into `*ExitError` and route specific exit codes deliberately rather than treating every non-zero exit as a hard error.
- **Unborn repo needs NO special handling here (simpler than log.go).** Unlike `rev-list --count HEAD`/`git log` (which exit 128 on an unborn repo and need the "unknown revision"/"does not have any commits yet" detection), `git diff --cached --quiet` on an unborn repo exits 0 (clean) or 1 (staged) just like a committed repo — verified. So stage.go has no unborn special-case at all.
- **Faithful port, no go-git, no shell (PRD §22.3/§19).** Every git call is a `[]string` run via `exec.Command` (the shipped `run` seam); `--cached`, `--quiet`, and `-A` are literal args, never interpolated into a shell string — shell injection is structurally impossible.

   - `_, err := g.run("diff", "--cached", "--quiet")`. (`--quiet` implies `--exit-code`: git exits 0 if the index matches HEAD — clean — and 1 if there are staged differences.)
