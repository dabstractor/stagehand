# PRP for P1.M8.T1.S1

## Objective

Ship six contributor-facing reference manifests providers/{pi,claude,gemini,opencode,codex,cursor}.toml at repo root, each a human-readable TOML copy of the manifest compiled into the binary by internal/provider/builtin.go (P1.M2.T3.S1), plus a golden-equivalence test in internal/provider/ that fails if any on-disk .toml ever decodes to a Manifest different from the compiled builtin — so the reference copies and the built-ins can never drift. These are REFERENCE copies (PRD §14); the tool still works with zero config because the built-ins are compiled in. DOCS: Mode A — each .toml documents its provider inline via a header comment + per-field comments.

## Context

## Goal

**Feature Goal**: Provide six human-readable, fully-documented `providers/*.toml` reference files (PRD §14) that are *golden-equivalent* to the six compiled-in provider manifests, guarded by an automated test that prevents drift between the on-disk copies and the binary.

**Deliverable**: (1) Six new files at repo root: `providers/pi.toml`, `providers/claude.toml`, `providers/gemini.toml`, `providers/opencode.toml`, `providers/codex.toml`, `providers/cursor.toml`. (2) One new Go test file `internal/provider/providers_golden_test.go` with a single table-driven test `TestProvidersTOML_MatchBuiltins` asserting each file decodes to `Builtins()[name]`.

**Success Definition**: `go test ./...` passes; the golden test fails (not skips) if any `providers/<name>.toml` is missing, malformed, or decodes to a Manifest that differs from the compiled builtin after nil/empty normalization; each file carries a header comment naming the provider and any TO-CONFIRM note (PRD §12.7.2, external_deps §B).

## Why

- Contributor reference & copy-paste source (parent task). Users learning the manifest schema (PRD §12.1) get six real, verified examples to copy when defining `[provider.<name>]` overrides (PRD §12.8).
- The compiled built-ins are authoritative (M2.T3.S1) but invisible inside the binary; these files make them inspectable/editable-as-template without a release.
- The golden test is a regression net: if someone edits `builtin.go` (or these files) the test forces the other side to be updated, catching silent divergence.

## Context Completeness Check

Passes "No Prior Knowledge": the authoritative field values are stated verbatim below (copied from `internal/provider/builtin.go`), the exact go-toml/v2 decode semantics are documented, and the golden-equivalence approach was prototyped and confirmed end-to-end in this repo.

```go
func TestProvidersTOML_MatchBuiltins(t *testing.T) {
    dir := filepath.Join("..", "..", "providers")
    names := sortedKeys(Builtins()) 
    for _, name := range names {
        name := name
        t.Run(name, func(t *testing.T) {
            raw, err := os.ReadFile(filepath.Join(dir, name+".toml"))
            if err != nil { t.Fatalf("missing providers/%s.toml: %v", name, err) } 
            var m Manifest
            if err := toml.Unmarshal(raw, &m); err != nil { t.Fatalf("decode %s.toml: %v", name, err) }
            want := Builtins()[name]
            m.Subcommand = normalizeEmpty(m.Subcommand); want.Subcommand = normalizeEmpty(want.Subcommand)
            m.BareFlags = normalizeEmpty(m.BareFlags); want.BareFlags = normalizeEmpty(want.BareFlags)
            if len(m.Env) == 0 { m.Env = nil }; if len(want.Env) == 0 { want.Env = nil }
            if !reflect.DeepEqual(m, want) {
                t.Errorf("providers/%s.toml decodes to a manifest that differs from the compiled builtin\n got  %+v\n want %+v", name, m, want)
            }
        })
    }
}

```

## Integration Points
None at runtime. These files are REFERENCE copies — they are NOT loaded by the config loader (M5.T3.S1) and NOT wired into the binary. User overrides still come from config-file `[provider.<name>]` tables. The only integration is the new golden test in internal/provider/.

## DOCS Impact (Mode A)
Each `.toml` file IS its own Mode A documentation: a header comment naming the provider/version/verification status + any TO-CONFIRM note, plus inline comments explaining non-obvious fields. No separate docs/ file is created by this item; changeset-level README/docs consolidation is a later M8 task (Mode B).

## Anti-Patterns to Avoid
- ❌ Do NOT hand-transcribe field VALUES from PRD text — transcribe from internal/provider/builtin.go (the PRD text predates the §C corrections; builtin.go has them applied).
- ❌ Do NOT emit raw `toml.Marshal` output as the file (no header comment, single-line arrays, all zero-value fields = unreadable). Hand-author for readability; correctness is enforced by the golden test, not by byte-identity.
- ❌ Do NOT add an `[env]` table to any file (nil-vs-empty map gotcha).
- ❌ Do NOT put retry_instruction in any file other than pi.
- ❌ Do NOT make the golden test t.Skip on a missing file — it must FAIL.
- ❌ Do NOT modify internal/provider/builtin.go (authoritative; out of scope).
- ❌ Do NOT wire these files into the runtime loader.
