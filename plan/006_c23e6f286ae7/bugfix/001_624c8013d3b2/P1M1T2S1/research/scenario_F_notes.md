## 2. The Load-Bearing Facts — all verified against live code

Scenario F's feasibility rests on FIVE code facts. Each is confirmed by reading the live source, so the
PRP can state them as facts (not risks for the implementer to rediscover):

### 2.1 Config defaults ENABLE decompose with the base stub config (no extras)

`internal/config/config.go` `Defaults()`:
```go
AutoStageAll: true,
Commits:      0,    // auto-decompose
Single:       false,
```
The base `writeStubConfig(t, stub, "")` TOML does NOT set `auto_stage_all`/`commits`/`single` → the
loaded config takes these defaults. `shouldDecompose(cfg, false, false)` (default_action.go:329, called
at :90) returns **true** when nothing is staged and the working tree is dirty. **No `extras` needed for
decompose activation.** (The contract's "if the loaded config does NOT default AutoStageAll to true… add
auto_stage_all = true" caveat is NOT triggered — the default IS true.)

### 2.2 Exactly one untracked file → the FR-M2b one-file shortcut (MESSAGE role only)

`internal/decompose/decompose.go:184-189`:
```go
if deps.Config.Commits == 0 && !deps.Config.Single {
    changedPaths, err := deps.Git.DiffTreeNames(ctx, baseTree, tStart)
    ...
    if len(changedPaths) == 1 {
        return runOneFileShortcut(ctx, deps, preRunHEAD, isUnborn, baseTree, tStart)
    }
}
```
With `Commits==0` (default) + `!Single` + exactly ONE changed path between `baseTree` and `tStart`
(one untracked file) → `runOneFileShortcut` is taken. Its doc comment (decompose.go:174): "a message
generated by the MESSAGE role: ZERO planner round-trip." So the stub's single-response
`STAGECOACH_STUB_OUT` satisfies the only agent call. **No planner role is invoked → no `[role.planner]`
extras needed.**

### 2.3 SetSnapshot(tStart) runs BEFORE the message-role call

`internal/decompose/decompose.go:165-189` ordering:
```
FreezeWorkingTree(ctx, baseTree)   // line 165 — resets .git/index to baseTree
lock.SetSnapshot(tStart)           // line 169 — publishes T_start to the lock file
... one-file check ...             // line 184-189
runOneFileShortcut → message-role stub → drain stdin → write marker → sleep
```
So when the stub writes the readiness marker (during message generation), `T_start` is **ALREADY
published** in the lock file. `waitForMarker` returning ⇒ holder holds the lock AND has published
`snapshot=T_start`.

### 2.4 The stub's marker/sleep ordering contract

`cmd/stubagent/main.go` `main()` order:
1. Drain stdin (deadlock guard).
2. Write readiness marker (`STAGECOACH_STUB_MARKER`) — **before** the sleep.
3. Sleep (`STAGECOACH_STUB_SLEEP_MS`).
4. Write stdout (`STAGECOACH_STUB_OUT`).
5. Exit.

So `waitForMarker` returning means: the holder has acquired the lock, frozen the working tree,
published `T_start`, called the message stub, the stub drained stdin + wrote the marker, and is now
**sleeping with the lock held**. The contention window is deterministically open.

### 2.5 The contention handler → Busy on the decompose path (the exact assertion strings)

`internal/cmd/default_action.go:241-256` `handleLockContention`:
```go
if snap := heldErr.Contents.Snapshot; snap != "" {
    contenderTree, werr := g.WriteTree(ctx) // index-read-only
    if werr == nil && contenderTree == snap {
        fmt.Fprintln(stderr, "nothing to do — an in-progress run already covers your staged changes.")
        return exitcode.New(exitcode.Success, nil) // exit 0
    }
}
fmt.Fprintf(stderr,
    "stagecoach: another stagecoach run is already in progress on %s (pid %s on %s). "+
        "Your newly-staged changes will remain staged — re-run stagecoach after it finishes. Lock: %s.\n",
    heldErr.Contents.Repo, heldErr.Contents.Pid, heldErr.Contents.Hostname, heldErr.Path)
return exitcode.New(exitcode.Busy, nil) // exit 5, SILENT
```
On the decompose path: `snap = T_start` (working-tree) ≠ `contenderTree = WriteTree() = baseTree`
(index reset to baseTree by FreezeWorkingTree, nothing staged → `git write-tree` returns `HEAD^{tree}`).
So the no-op arm fails → **Busy(5)** with the "already in progress" message; "nothing to do" does NOT
appear.

**Exact assertion substrings:**
- Busy path contains: `"already in progress"` ✓
- No-op path contains: `"nothing to do"` — must NOT appear on the decompose path ✓

## 3. Assertion Robustness — why ExitCode==5 is timing-proof

The assertion `res2.ExitCode == 5` holds **regardless of the microsecond timing** of `SetSnapshot` vs
the contender's read:

| Snapshot state when contender checks | Code path | Result |
|---|---|---|
| `snap = T_start` (published) | `WriteTree() = baseTree ≠ T_start` → no-op arm fails → Busy | **5** |
| `snap = ""` (race — not yet published) | `snap != ""` is false → no-op arm SKIPPED → Busy | **5** |

Either way → **5**. This makes scenario F robust to scheduling jitter while still pinning the documented
behavior (decompose double-run = Busy, NOT 0). The "must NOT contain 'nothing to do'" assertion is
similarly robust: the no-op arm either fails the SHA compare (→ Busy, no "nothing to do") or is skipped
entirely (→ Busy, no "nothing to do").

## 4. The Scenario F Skeleton (verified against the scenario-B pattern + live code)

Reuses the canonical two-process contention skeleton (goroutine + `waitForMarker` + channel drain)
verbatim from scenario B. The ONLY differences from B: (a) ONE untracked file NOT staged (decompose
activation) instead of a staged file; (b) asserts `ExitCode == 5` (Busy) + "already in progress" + NOT
"nothing to do", instead of `ExitCode == 0` + "nothing to do".

```go
t.Run("F_DecomposeAccidentalDoubleRun_Busy", func(t *testing.T) {
    repo := newRepo(t)
    seedCommit(t, repo, "readme.md", "init")
    writeFile(t, repo, "feature.txt", "new work\n") // ONE untracked file, NOT staged → decompose activates

    readiness := t.TempDir() + "/ready.marker"
    holderEnv := stubEnv(map[string]string{
        "STAGECOACH_STUB_OUT":      "feat: add feature",
        "STAGECOACH_STUB_MARKER":   readiness,
        "STAGECOACH_STUB_SLEEP_MS": "4000",
    })

    resCh := make(chan e2eResult, 1)
    go func() { resCh <- runStagecoach(t, bin, repo, cfg, holderEnv, "--provider", "stub") }()
    waitForMarker(t, readiness, 10*time.Second) // holder: lock held, T_start published, message-gen sleep

    // Contender: same dirty tree, still nothing staged → handleLockContention:
    //   WriteTree() = baseTree ≠ snap(T_start) → Busy(5). "nothing to do" must NOT appear.
    contenderEnv := stubEnv(map[string]string{"STAGECOACH_STUB_OUT": "feat: add feature"})
    res2 := runStagecoach(t, bin, repo, cfg, contenderEnv, "--provider", "stub")
    if res2.ExitCode != 5 {
        t.Fatalf("contender exit = %d, want 5 (Busy) — decompose no-op fast path is structurally impossible; stderr:\n%s", res2.ExitCode, res2.Stderr)
    }
    if !strings.Contains(res2.Stderr, "already in progress") {
        t.Errorf("stderr missing busy message; got:\n%s", res2.Stderr)
    }
    if strings.Contains(res2.Stderr, "nothing to do") {
        t.Errorf("decompose path must NOT hit the no-op fast path; got:\n%s", res2.Stderr)
    }

    res := <-resCh // drain holder
    if res.ExitCode != 0 {
        t.Fatalf("holder exit = %d, want 0; stderr:\n%s", res.ExitCode, res.Stderr)
    }
})
```

**`bin`, `stub`, `cfg` come from the `TestE2ELockContention` outer scope** (lines just below the func
declaration): `bin := buildStagecoach(t)`, `stub := buildStub(t)`, `cfg := writeStubConfig(t, stub, "")`.
So scenario F does NOT redeclare them — it reuses the shared `cfg` (the architecture skeleton's local
`cfg := writeStubConfig(...)` line would shadow; drop it and use the outer `cfg`). This matches how
scenarios A–E are written (they all use the outer-scope `bin`/`cfg`).
